# Invariants

> Properties that must ALWAYS hold. A violation is a bug, by definition.
> Scope: behavior that must survive the zimgx (Zig) → imgx (Rust) rewrite. These are the
> spec for TDD during the port — see /Users/christopherw/.claude/plans/recursive-meandering-crescent.md.
> Each invariant names the Zig source it's captured from; the Rust port must have a test
> at the same granularity, not just a manual read-through.

## System invariants

### INV-1: A transform-params cache key is a pure, deterministic function of the parsed parameters, independent of input order

- **Always:** For any two parameter strings that parse to the same `TransformParams` value (regardless of key order in the input string), `toCacheKey` produces byte-identical output. Fields are written in a fixed order: `w, h, q, f, fit, g, sharpen, blur, dpr, rotate, flip, brightness, contrast, saturation, gamma, bg, metadata, trim, anim, frame`. Optional fields at their default/unset value are omitted entirely, not written with a default value.
- **Because:** `src/transform/params.zig:359-383` (`writeCacheKey`) always visits fields in this literal order regardless of parse order, and guards optional fields with `if (x) |v|` / `if (x != default)`.
- **If violated:** Two functionally-identical requests get different cache keys → cache poisoning/duplication. Worse, an existing R2-cached variant (written by the old Zig binary) becomes unreachable under the Rust binary because the key it would compute differs — a silent full cache invalidation of the L2 tier in production.

### INV-2: Animated-image resize with `fit=cover` never corrupts frame boundaries

- **Always:** When resizing a multi-frame (animated) image with `fit=cover` and both `w` and `h` set, every output frame has identical height (`resized_height`), and `page-height` metadata on the output stack always evenly divides the total stack height. The number of frames in the output never exceeds `min(source_frames, max_frames, 256)`.
- **Because:** `vips_thumbnail_image`'s built-in crop corrupts frame boundaries on vertically-stacked animated buffers (upstream libvips bug #2668). `src/transform/pipeline.zig:212-260+` (resize-without-crop, then either one `extract_area` for horizontal-only crop, or a per-frame crop+`arrayjoin(across=1)` loop capped at 256 frames for vertical crop) is a deliberate, hard-won workaround. Skipping it reintroduces the bug.
- **If violated:** Encoded animated output has visibly wrong/sheared frames, or the GIF/WebP encoder reads past a frame boundary — this exact class of bug previously caused a SIGSEGV (see INV-3).

### INV-3: GIF encoding never receives inconsistent page-height/n-pages metadata

- **Always:** Before `vips_gifsave_buffer` is called on an image carrying `n-pages > 1`, `page-height` metadata evenly divides the image's total height. If it does not, `page-height` is reset to the full image height and `n-pages` is reset to 1 (degrade to a single-frame GIF) before encoding.
- **Because:** `src/transform/pipeline.zig` GIF encode path — a documented regression test reproduces a real prior SIGSEGV in libvips' GIF encoder caused by stale/inconsistent page metadata after cropping.
- **If violated:** Process crash (SIGSEGV) serving a specific animated-GIF transform request — a live availability incident, not just a wrong-output bug.

### INV-4: A request path can never resolve to a filesystem/origin path outside the intended root

- **Always:** `resolve(path)` returns `.not_found` for any path containing a null byte, a literal `..` traversal segment, a percent-encoded traversal/null-byte sequence (`%2e`, `%2E`, `%2f`, `%2F`, `%00`), or an embedded absolute path (`//...`). This holds both for `sanitizePath` called directly and transitively through `resolve`.
- **Because:** `src/router.zig:107-189` (`sanitizePath`, `containsTraversal`, `containsEncodedTraversal`) is the only gate between an untrusted client-supplied path and the origin fetch key / R2 object key.
- **If violated:** Path traversal — the proxy could be tricked into fetching or exposing origin content outside the intended prefix (SSRF-adjacent / origin data exposure).

### INV-5: Every image request uses imgx's fixed `image/<OPTIONS>/<SOURCE-IMAGE>` convention — a fixed prefix, then an OPTIONS segment, then the source path

- **Always:** For a cleaned path, the request must start with the literal prefix `image/`; anything else is `.not_found` (the old trailing-options shape, `/<image-path>/<transforms>`, is retired — this was a conscious, full breaking migration, not an additive route). The segment immediately following the prefix is treated as OPTIONS: if it contains `=`, it is the transform string and everything after it (which may itself contain `/`) is the source image path — unless that remainder is empty, which is `.not_found` (options with no image to apply them to). **Decision:** if that segment does *not* contain `=`, imgx treats it as the start of the image path instead of rejecting the request (relaxing Cloudflare's "at least one parameter" rule, the closest prior art for this URL shape) so a transform-less passthrough request (e.g. `/image/photos/cat.jpg`) still resolves — image path `photos/cat.jpg`, no transform. A path like `image/a=1/b=2` yields transform `a=1`, image path `b=2` — the first segment after the prefix is the only one ever treated as a transform, never a later one.
- **Because:** `crates/imgx/src/router.rs`'s `resolve` (originally ported from `src/router.zig:57-97` with 17 router tests for the old trailing-options shape; migrated to Cloudflare's options-first, fixed-prefix convention as a deliberate breaking change so imgx's canonical URLs are byte-for-byte compatible with Cloudflare's documented format). Tests updated in place to the new shape rather than kept alongside stale old-shape tests.
- **If violated:** URLs stop being interchangeable with Cloudflare's format (defeating the purpose of the migration), or — if the no-`=`-segment passthrough decision above were dropped silently — transform-less requests that used to resolve would start 404ing.

### INV-6: Every `TransformParams` field with numeric or float bounds rejects out-of-range, NaN, and Inf values at parse/validate time — vips never sees an invalid value

- **Always:** `width`/`height` ∈ [1,8192]; `quality` ∈ [1,100]; `dpr` ∈ [1.0,5.0]; `sharpen` ∈ [0.0,10.0]; `blur` ∈ [0.1,250.0]; `brightness`/`contrast`/`saturation` ∈ [0.0,2.0]; `gamma` ∈ [0.1,10.0]; `trim` ∈ [1.0,100.0]; `frame` ≤ 999. Any float parse of `nan`/`inf`/`-inf` is rejected before it can reach `validate()`.
- **Because:** `src/transform/params.zig:314-347` (`validate`) plus `parseF32`'s explicit `isNan`/`isInf` guard at line 471-475. This is the sole gate before values are handed to libvips C calls that do not themselves validate ranges (e.g. a NaN sigma into `vips_gaussblur` is undefined behavior territory).
- **If violated:** Malformed/malicious query strings reach libvips FFI calls with unchecked values — crash or undefined behavior risk at the FFI boundary, not just a bad HTTP response.

### INV-7: Format negotiation priority is fixed and total — it always resolves to a concrete format, never leaves it ambiguous

- **Always (static):** explicit non-auto format always wins outright. Otherwise: alpha-present source → avif > webp > png > jpeg(last resort); no-alpha source → avif > webp > jpeg > png. Empty/absent Accept header → jpeg. **Always (animated):** explicit animation-capable format (webp/gif) wins; explicit non-animation-capable format → `null` (caller must degrade to static); auto-negotiate → webp > gif > `null`.
- **Because:** `src/transform/negotiate.zig:108-185`, 30 tests covering wildcard, q=0 disabling, alpha/no-alpha branches, malformed headers.
- **If violated:** Either a format the client didn't ask for and can't render (broken image in browser), or non-deterministic format selection breaking cache-key/response consistency for identical requests.

### INV-8: The server refuses to start with an invalid configuration rather than running with a partially-bad config

- **Always:** `validate()` rejects (does not clamp or silently ignore): port 0; either timeout = 0; `max_width`/`max_height` = 0; `default_quality` outside [1,100]; HTTP origin with a `base_url` that doesn't start with `http://` or `https://`; R2 origin with any of endpoint/access_key_id/secret_access_key/bucket_originals/bucket_variants empty; `server.max_connections`/`max_request_size` = 0; `cache.max_size_bytes`/`default_ttl_seconds` = 0 when the cache is enabled; `transform.max_frames`/`max_animated_pixels` = 0.
- **Because:** `src/config.zig:203-233` (Zig original) / `crates/imgx/src/config.rs`'s `validate()` (Rust — extended beyond the Zig original's coverage after an audit found several loaded-and-validated-nowhere fields; see INV-11/INV-12 below for two of them). Config is loaded once at startup and validated before the server binds a listener.
- **If violated:** Server starts in a broken state (e.g. no valid origin configured, or a cache/animation limit that silently disables the subsystem it gates) and fails every request at runtime instead of failing fast at deploy time — much harder to diagnose in production (looks like an outage, not a config error).

### INV-9: HTTP error responses always carry the documented status code and JSON envelope shape

- **Always:** Every domain error maps to exactly one of {400, 404, 413, 422, 500, 502, 504} via the named constructors, and serializes to `{"error":{"status":N,"message":"...","detail":"..."}}` with the `detail` key entirely absent (not `null`) when no detail was given.
- **Because:** `src/http/errors.zig` — status codes are fixed per constructor (`badRequest`→400, `notFound`→404, `payloadTooLarge`→413, `unprocessableEntity`→422, `internalError`→500, `badGateway`→502, `gatewayTimeout`→504), and `toJsonResponse` branches on `self.detail` being null to omit the field.
- **If violated:** Callers/clients that pattern-match on the error envelope shape (e.g. checking for absence of `detail` vs. `detail: null`) break; monitoring/alerting keyed on status codes miscounts error classes.

### INV-11: A source image's decoded pixel budget is checked before any resize/effect/encode work touches it

- **Always:** Immediately after the first frame of a source image is probed, `current.width() * current.height()` is compared against `transform.max_pixels`. If it exceeds the budget, the transform is rejected (`TransformError::ExceedsMaxPixels`) before `vips_thumbnail_image`, any effect filter, or an encoder ever runs on it.
- **Because:** `crates/imgx/src/transform/pipeline.rs`'s `transform()` — `IMGX_TRANSFORM_MAX_PIXELS` was loaded and validated by `config.rs` from the start of the Rust port but never actually read anywhere else, found during a post-launch audit. This is distinct from INV-2/`max_animated_pixels`, which budgets the *total* pixels across an animated stack — INV-11 catches an oversized *first frame* even on a source that turns out not to be animated at all.
- **If violated:** A crafted image with a small compressed size but enormous declared dimensions gets fully decoded into raw pixel memory with no cap — a classic decompression-bomb DoS against server memory.

### INV-12: The origin fetcher never buffers an unbounded or over-limit response body into memory

- **Always:** A declared `Content-Length` over `origin.max_size` is rejected before any body bytes are read. A response streamed without (or lying about) `Content-Length` is capped while streaming — accumulated bytes are checked after every chunk, and the fetch aborts the moment the running total exceeds `max_size`, rather than reading the full body first and checking afterward.
- **Because:** `crates/imgx/src/origin/fetcher.rs`'s `fetch()` — the original implementation called `resp.bytes().await` (buffer everything) before ever comparing the length against `max_size`, found during the same audit as INV-11. A compromised or misbehaving origin returning an unbounded body would be fully buffered per in-flight request regardless of the configured limit.
- **If violated:** Unbounded per-request memory growth driven entirely by origin response size — a resource-exhaustion DoS surface independent of anything the client controls.

### INV-13: A failed transform is never cached under the same key a successful transform of the same request would use

- **Always:** When `pipeline::transform()` fails (decode error, unsupported codec, panic) and the server falls back to serving the raw origin bytes, that fallback response is *not* written to the cache. The `cache_key` for a given `(path, transform_string, format)` is only ever populated by an actually-successful transform.
- **Because:** `crates/imgx/src/server.rs`'s `handle_image_request` — the original fallback wrote the raw passthrough bytes to `cache_key` (the same key a successful transform of the identical request would use), so a transient failure (e.g. a brief OOM, a runtime missing a codec plugin — see the AVIF incident that prompted this audit) would poison every subsequent identical request with the wrong (untransformed) response for the full `default_ttl_seconds`, long after the underlying cause recovered.
- **If violated:** A transient, already-recovered failure keeps silently serving wrong output for up to an hour (default TTL) after the fact — the cache actively works against recovery instead of being neutral to it.
- **Extended (Cloudflare parity gap 7, `docs/CLOUDFLARE_PARITY.md`):** an opt-in per-request `onerror=redirect` parameter (`TransformParams::onerror`, matching Cloudflare's own `onerror=redirect`) additively changes the failure-path *response* — a `302` to the origin image URL instead of the raw-bytes fallback, for `Http` origins only. This does NOT weaken the guarantee above: the default (`onerror` unset) behavior is unchanged, and neither the redirect nor the raw-bytes fallback is ever written to `cache_key`. See `crates/imgx/src/server.rs`'s `handle_transform_failure`.

### INV-14: An arbitrary attacker-influenced URL is never fetched without every SSRF guard applying, and is never fetched at all unless explicitly opted into

- **Always:** By default (`IMGX_ALLOW_REMOTE_SOURCES=false` and `IMGX_ALLOW_DRAW_OVERLAYS=false`, both defaults), imgx never attempts a network fetch to a URL named in client-controlled request data — a request naming an absolute `http(s)://` source or a `draw[].url` overlay is rejected with `403` before any network call. When either flag is enabled, every fetch through `origin::RemoteFetcher` (`crates/imgx/src/origin/remote.rs`) enforces, unconditionally and together: (1) scheme allowlist (`http`/`https` only, on the initial URL and every redirect target); (2) DNS-resolution-time rejection of non-globally-routable resolved addresses (private/RFC1918, loopback, link-local, RFC6598 CGNAT, `0.0.0.0/8`, multicast, broadcast, and IPv6 equivalents including IPv4-mapped-IPv6 literals), checked *after* real resolution so DNS rebinding cannot bypass it; (3) a separate direct check rejecting literal-IP-address hosts in the same non-globally-routable ranges, since standard HTTP connectors skip DNS resolution entirely for a URL whose host is already an IP literal — the resolution-time check in (2) would never even run for that case; (4) a capped (3), re-validated redirect chain — each hop re-checked against (1)–(3), not just the initial URL; (5) the same streaming size cap as the origin fetcher (INV-12), never buffering an unbounded/over-limit body; (6) the same per-request timeout as the origin fetcher.
- **Because:** `crates/imgx/src/origin/remote.rs` — Cloudflare parity gap 2 (arbitrary remote-URL main-image sources) and gap 11 (draw-overlay remote fetch), both deliberately deferred twice before this implementation specifically because an unguarded fetch of a client-influenced URL is a textbook SSRF vector against internal services (cloud metadata endpoints, internal APIs, other containers on the same host/network). Guard (3) was found during implementation, not designed upfront: hyper's connector (which both `reqwest` and this fetcher build on) shorts-circuits DNS resolution for an already-literal IP host, so a URL pointed directly at e.g. `169.254.169.254` would have silently bypassed a resolution-time-only guard.
- **If violated:** The proxy becomes a confused-deputy SSRF proxy — an attacker who controls a `draw[].url` or a request's source-image segment could make the imgx process issue requests to internal-only services (cloud metadata endpoints, internal admin APIs, other hosts on a private network) that the attacker could not otherwise reach directly, using imgx's network position and any credentials/trust that position implies.
- **Note:** These guards mitigate, but do not eliminate, SSRF risk against internal services when either flag is enabled — they are a defense-in-depth network-layer check, not a substitute for actual network segmentation (firewalling imgx away from sensitive internal services regardless of what the application layer permits). An operator enabling either flag is opting into fetching attacker-influenced URLs; see `apps/docs/src/pages/configuration.mdx` for the operational warning.

## Enforceable invariants

Each maps to a concrete check that must exist in the Rust port with equivalent coverage.

- [x] **INV-1 (cache key determinism/order-independence):** enforced by `params.zig` tests `"cache key is deterministic"`, `"cache key order is canonical regardless of parse order"`, `"cache key differs when params differ"`, `"cache key omits default metadata"`, `"cache key includes non-default metadata"`, `"cache key omits anim when auto"`, `"cache key includes anim when not default"`, `"cache key includes frame when set"`. Port all of these verbatim in `transform/params.rs`, plus add a table of literal expected key strings for a handful of param combos (byte-for-byte parity check against the Zig implementation, not just internal self-consistency).
- [x] **INV-2 (animated cover-resize frame integrity):** enforced only implicitly today via manual/fixture testing against `test/fixtures/loading.gif` (12 frames, 128×128) — **GAP-adjacent**: no explicit Zig test asserts "every output frame has identical height" as a property. Rust port: add a property/golden test that decodes the output stack and asserts `total_height % n_pages == 0` and (if per-frame decoding is feasible with the chosen output codec) that frame boundaries land where expected.
- [x] **INV-3 (GIF page-height safety check):** enforced by the page-height-corruption regression test in `pipeline.zig` (explicitly reproduces the prior SIGSEGV scenario). Port verbatim in `transform/pipeline.rs`.
- [x] **INV-4 (path traversal rejection):** enforced by `router.zig` tests — `"path traversal is rejected by sanitizePath"`, `"null byte is rejected..."`, `"sanitizePath rejects embedded absolute path"`, `"sanitizePath rejects dot-dot at start/end"`, `"sanitizePath rejects percent-encoded dot traversal"` (upper+lowercase), `"sanitizePath rejects encoded null byte"`, `"sanitizePath rejects encoded slash"`. Port every one verbatim in `router.rs`. **Extended (audit finding):** `origin/source.rs`'s `build_url` had no traversal check of its own, relying entirely on the router having already sanitized `image_path` upstream — inconsistent with `cache/r2.rs`'s own defense-in-depth (INV-10). Added an equivalent `..`-rejection check directly in `build_url`, tested by `build_url_rejects_traversal_sequence`/`build_url_rejects_embedded_traversal_sequence`, so the origin-fetch trust boundary doesn't depend solely on a caller having sanitized its input correctly.
- [x] **INV-5 (fixed `image/<OPTIONS>/<SOURCE>` URL scheme):** enforced by `router.rs` tests `image_with_transforms`, `image_without_transforms`, `nested_path_with_transforms`, `nested_path_without_transforms`, `multiple_transform_like_segments_only_first_is_treated_as_transform`, `path_with_only_transforms_and_no_image_path_returns_not_found`, `options_segment_with_equals_but_empty_remainder_returns_not_found`, `transform_detection_segment_with/without_equals_...`, `path_without_image_prefix_returns_not_found`, `bare_image_prefix_returns_not_found`. **Extended (breaking migration):** the old trailing-options shape's tests (formerly ported verbatim from `router.zig`) were rewritten in place to the new options-first, fixed-prefix shape rather than kept as a parallel deprecated route — see the decision recorded in the invariant text above. **Extended (dropped `cdn-cgi/`):** imgx originally mirrored Cloudflare's exact `cdn-cgi/image/` prefix for drop-in compatibility, then deliberately dropped the `cdn-cgi/` segment as a CGI-bin-era relic irrelevant outside Cloudflare's own edge, keeping `image/<OPTIONS>/<SOURCE>` — a breaking URL change, not an additive one.
- [x] **INV-6 (parameter bounds + NaN/Inf rejection):** enforced by ~20 boundary tests in `params.zig` (`"width boundary values"`, `"*_bounds_validation"` for sharpen/blur/dpr/brightness/contrast/saturation/gamma/trim, `"parse rejects nan values"`, `"parse rejects inf values"`, `"frame validation rejects large values"`). Port verbatim; this set is the primary FFI-safety gate so treat any gap here as high-severity.
- [x] **INV-7 (negotiation priority):** enforced by 30 tests in `negotiate.zig` covering static and animated paths, alpha/no-alpha, wildcards, q=0, malformed headers. Port verbatim.
- [x] **INV-8 (config validation is a hard gate):** enforced by `config.zig` tests `"validate rejects port 0"`, `"...empty base_url"`, `"...base_url without http scheme"`, `"...base_url with file scheme"`, `"...quality 0/101"`, `"...zero request_timeout_ms/origin timeout_ms"`, `"...zero max_width/max_height"`. Port verbatim; also port `"loadFromEnv with no env vars returns defaults"` since env-var-prefix renaming (`ZIMGX_`→`IMGX_` with fallback, per the approved plan) is exactly the kind of change that silently breaks this invariant if done carelessly. **Extended (audit finding):** `config.rs`'s `validate()` didn't reject `server.max_connections`/`max_request_size` = 0 or `transform.max_frames`/`max_animated_pixels` = 0 (each silently disables the subsystem it gates rather than failing fast), nor `cache.max_size_bytes`/`default_ttl_seconds` = 0 while the cache is enabled. Added `validate_rejects_zero_max_connections`, `validate_rejects_zero_max_request_size`, `validate_rejects_zero_cache_max_size_bytes_when_cache_enabled`, `validate_rejects_zero_cache_ttl_when_cache_enabled`, `validate_accepts_zero_cache_limits_when_cache_disabled`, `validate_rejects_zero_max_frames`, `validate_rejects_zero_max_animated_pixels` in `config.rs`.
- [x] **INV-9 (error status/envelope shape):** enforced by `errors.zig` tests for every constructor plus `"toJsonResponse with detail"` / `"...without detail omits detail field"`. Port verbatim — in Rust this likely becomes a `Serialize` impl or `serde_json::json!` builder; add a test asserting the `detail` field is skipped (`#[serde(skip_serializing_if = "Option::is_none")]`) rather than emitted as `null`, since that's an easy accidental regression when moving from hand-rolled `bufPrint` to `serde`.
- [x] **INV-10 (R2 cache key sanitization prevents traversal into arbitrary R2 objects):** `src/cache/r2.zig`'s `sanitizeKey` (`|`→`/`, collapse `//`, reject `..`) returned an empty string on violation with callers never checking it — a latent bug in the original Zig code. Fixed in the Rust port: `cache/r2.rs`'s `sanitize_key` returns `Option<String>`, and every caller (`get`/`put`/`delete`) handles the `None` case explicitly (treated as a miss/no-op, never issuing a malformed-key S3 request). Tested by `sanitize_key_rejects_traversal_sequences` and `get_treats_traversal_key_as_a_miss_not_a_request`.
- [x] **INV-11 (source pixel budget checked before decode):** `crates/imgx/src/transform/pipeline.rs`, tested by `transform_rejects_source_exceeding_max_pixels`/`transform_accepts_source_within_max_pixels`.
- [x] **INV-12 (origin fetch never buffers past max_size):** `crates/imgx/src/origin/fetcher.rs`, tested by `fetch_rejects_declared_content_length_over_max_size_without_reading_body`, `fetch_rejects_streamed_body_over_max_size_even_without_content_length`, `fetch_accepts_body_within_max_size`.
- [x] **INV-13 (failed transform never poisons the success cache key):** `crates/imgx/src/server.rs`, tested by `transform_failure_fallback_is_never_cached`.
- [x] **INV-14 (SSRF guards on any attacker-influenced fetch):** `crates/imgx/src/origin/remote.rs`, tested by `rejects_ipv4_*`/`rejects_ipv6_*`/`accepts_*` (pure unit tests of the private-address predicate), `fetch_rejects_url_resolving_to_loopback`/`fetch_rejects_a_redirect_to_a_private_address_when_guard_is_enabled` (end-to-end, real production constructor), `fetch_rejects_non_http_scheme_without_a_network_call`/`fetch_rejects_file_scheme`/`fetch_rejects_gopher_scheme`, `fetch_rejects_a_redirect_chain_longer_than_the_cap`/`fetch_rejects_a_redirect_to_a_non_http_scheme`, and the size-cap tests mirroring INV-12. Default-off wiring tested by `crates/imgx/src/server.rs`'s `remote_source_request_is_rejected_by_default_with_403_and_no_network_call`/`draw_overlay_request_is_rejected_by_default_with_403`.

## Assumptions & non-invariants

- **Encoded output bytes are NOT invariant across the rewrite.** libvips encoder versions differ, and Wyhash→xxh3 changes the ETag. Parity testing (plan phase 8) compares status codes, headers, cache keys, and image *dimensions* — never raw output bytes.
- **The 256-thread-pool / atomic-connection-counter backpressure mechanism is NOT preserved as-is.** The approved plan deliberately replaces it with an async `tokio`/`axum` model using a `Semaphore` gated on `spawn_blocking` permits. The *invariant* that survives is "the server rejects new work under saturation rather than unbounded-queueing it," not the specific mechanism.
- **`allocator.zig`'s `RequestArena`, R2Cache's "last-fetched-buffer" ownership hack, and threadlocal error-formatting buffers are Zig-specific workarounds, not invariants.** They exist because of Zig's manual memory management; Rust's ownership model makes them unnecessary. Do not port them — porting a workaround whose underlying problem no longer exists is itself a bug (unnecessary complexity, and a plausible source of new bugs).
- **Cache-key byte-for-byte preservation (INV-1) is a policy choice, not a law of nature** — the approved plan chose to preserve it so existing R2-cached variants remain valid across the binary swap. If that decision is ever revisited (e.g. deliberate cache-busting rewrite), INV-1 is the invariant to explicitly relax, and it should be a conscious, documented decision, not a side effect of a refactor.
